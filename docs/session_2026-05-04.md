Session notes: Round-2 & follow-up (2026-05-04)

Major finding
- FPGA pipeline is end-to-end correct: A RX -> FPGA parser/forwarder -> B TX -> PC2. SignalTap captures show 3-of-3 IPv4 frames forwarded with identical first-16 bytes on A and B.
- Forwarded frames were mDNS (UDP/5353) from the Mac, not the demo UDP/80 frames.

What changed in this session
- scripts/inspect_signaltap_csv.py: fixed trailing-X logic (now picks last non-X per column) and improved diagnosis output.
- scripts/signaltap_capture_force.tcl & scripts/make_anytrig_stp.py: added force-capture/relaxed-trigger helpers so CSV is exported even when the trigger doesn't fire.
- scripts/rule_demo_sender.py: default changed to broadcast (dst_mac=ff:ff:ff:ff:ff:ff, dst IP=192.168.1.255) to bypass macOS multicast routing; --dst-mac still available.
- rtl/eth_if/ethernet_controller_adapter.v: added SHAR write on init and S0_CR clear poll after RECV to avoid W5500 RX corruption.
- rtl/eth_if/w5500_macraw_tx_adapter.v: mirrored SHAR write for consistency with A.
- tb models: confirmed S0_CR and SHAR behaviors are handled; Questa regression re-run passed for all affected tests.
- Quartus: full compile started (background). Flash expected after compile finishes.

Why the demo frames were missing
- SignalTap round-2 proves IPv4 frames reach A but frames_demo_match==0 and frames_udp_dport80==0.
- The Mac was emitting mDNS (UDP/5353) multicast, not the demo UDP/80 frames.
- Two possibilities found earlier: (A) sender script on PC1 was not using the real interface or macOS was steering multicast via lo0 / PF blocked Scapy; (B) a wiring/cable or W5500 physical link issue. Evidence points to PC1-side behavior.

Temporary mitigation applied
- Sender default switched to broadcast (192.168.1.255) to avoid macOS multicast routing issues. User restarted the sender on PC1 and verified tcpdump saw UDP/80 broadcast packets leaving en0.

Firmware/RTL hardening applied
- Added SHAR write during adapter init (A and B) to avoid MAC-layer RX problems.
- Added S0_CR command-clear polling after RECV to ensure RECV completion before further SPI accesses.
- Questa regression: all relevant tests pass after changes.

Observed hardware state & recommended bench checks
- After restart, A adapter remained in ST_RX_POLL with zero RX size; W5500 A's RJ45 link/activity LEDs and physical cable reseating recommended.
- Recommended PC1 checks: run `sudo tcpdump -i en0 -nn -e -c 5 udp port 80` while sender runs. If tcpdump shows no packets, the Mac is not putting UDP/80 on en0.

Next steps (short)
1. If PC1 shows UDP/80 on the wire (tcpdump), reflash the new SOF after Quartus compile completes and recapture with `signaltap_capture_force.tcl`.
2. If PC1 shows nothing, investigate macOS PF/firewall or use broadcast/sender flags.
3. Optionally add a PHY link-status probe (SignalTap + recompile) to observe link/activity inside the FPGA if physical LEDs are inconclusive.
